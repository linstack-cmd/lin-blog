Every operational update system I've tried fails for the same reason: it either spams me constantly, or it goes stale and quietly lies. I got tired of both, so I built something stupidly simple that actually works.

I run an AI agent — Lin, on OpenClaw — for day-to-day ops. Checking service status, tracking platform quirks, answering "what changed?" when I sit down after a few hours away. Early on, I hit a predictable wall: **my agent's memory of current state is not a source of truth**.

Not because the model is bad. Because context resets, sessions roll over, and stale facts sound dangerously fresh when phrased with confidence. I watched it happen enough times to stop trusting verbal status reports entirely. So I built a watchlist.

> **TL;DR**
>
> I keep a `WATCHLIST.md` with only active items worth tracking.
> When I ask for updates ("any updates I need to know?", "what should I know?"), the agent is required to re-read that file *at answer time*.
> Result: fewer stale answers, less noise, faster recovery after resets.

## The problem I kept hitting

Here's what would happen. I'd ask Lin "anything I should know?" and get a confident, well-structured answer — pulled from session context that was three hours old. The answer *sounded* right. That's what made it dangerous.

Two real examples from my setup:

- **GPT-5.4 availability under my current auth mode.**
  I connect OpenAI via Codex OAuth (`openai-codex/*`). That means `openai-codex/gpt-5.3-codex` is available, while standard `openai/*` models (like `openai/gpt-5.4`) aren't selectable through this auth path. Lin once told me otherwise because an older session had different context.
- **Telegram duplicate-message behavior with streaming enabled.**
  Streaming is useful, but my environment still reproduces duplicates when it's on. I keep streaming off for reliability. After a session reset, Lin cheerfully recommended turning it on — old context, new session, wrong answer.

Both cases share the same failure mode: the agent answers from memory that was true earlier but isn't true now, and nothing in the interaction signals that the information is stale. I needed answers to come from somewhere more durable than context.

## What I built: `WATCHLIST.md` + one hard rule

I landed on a two-part system after trying a couple of more elaborate approaches that all felt like overkill.

### 1) A small source-of-truth file

`WATCHLIST.md` lives in my workspace. I track only items that matter *right now* — each with current status and my interpretation. Not a giant log, not a running diary. Just the active operational state I want my agent to know about.

Example structure:

```md
### Item
- What to watch
- Latest status (date)
- Interpretation / next move
```

I keep it to a handful of entries. The moment something resolves or stops being relevant, I prune it.

### 2) A mandatory "re-read before reply" rule

When I ask for updates — "any updates?", "what should I know?", "catch me up" — the agent **must read `WATCHLIST.md` fresh before answering**.

Not optional. Not "if it seems relevant." Every single time.

This is the part that actually matters. One strict retrieval rule turns status replies from vibes-from-context into grounded responses from a file I control.

## Why it works (and why it's embarrassingly simple)

The core insight took me longer to accept than it should have: **file-backed answers beat memory-backed answers**.

Session resets don't matter because the watchlist persists on disk. I only get updates on items I've chosen to track, so the noise floor stays low. And if I ever doubt what the agent told me, I can open the file and see exactly what it knows — and what it doesn't.

The whole system is auditable by design. After debugging enough "where did the agent get *that*?" moments, I've come to value auditability more than sophistication. I can trace every status answer back to a line in a markdown file. That's the kind of boring I want.

## Tradeoffs (worth acknowledging)

- **Manual curation is required.** If I don't update the file, it rots.
- **Not real-time monitoring.** It reflects known state, not live telemetry.
- **Trigger phrase ambiguity.** Natural language is fuzzy; the guardrail needs "similar intent" judgment from the agent.
- **Scaling limit.** One file works great for a handful of items. Bigger systems would need categories or split files.

For my lean operator workflow, the tradeoff is excellent: tiny upkeep cost for a big trust upgrade.

### Limitations

I want to be honest about what this system *doesn't* do, because the simplicity cuts both ways.

**It's not real-time by design.** The watchlist reflects known state as of the last time I (or the agent) updated it. There's no live telemetry, no websocket feed, no continuous polling. Between checks, the world can change and the file won't know.

**It's a pull model.** Nothing happens until I ask — or until a scheduled heartbeat triggers a read. If I don't ask "any updates?" for three days, the watchlist just sits there. I can mitigate this with cron-triggered checks, but out of the box it's purely on-demand.

**Staleness accumulates between checks.** Even with the fresh-read guardrail, the *file contents* can drift from reality. The guardrail ensures the agent reads the file honestly; it doesn't ensure the file itself is current. That's on me.

**Manual maintenance is a real cost.** Every tracked item needs me to write it, update it, and eventually prune it. For three to five items, that's trivial. At twenty items, it's a chore I'd start skipping — and a skipped update is a stale update.

**There's no automatic incident detection or escalation.** If a service goes down at 2 AM, this system won't page me. It won't even notice until the next check. This is a briefing layer, not a monitoring stack — I still need actual alerting for anything urgent.

These are acceptable constraints for my use case, but they're worth knowing before you adopt the pattern.

## When to reach for this pattern

I'd recommend this if your agent answers status questions, runs across multiple sessions or resets, and needs to be right more often than it needs to be flashy.

It doesn't replace monitoring or alerts. It complements them — a clean briefing layer, a single file the agent is required to consult before it opens its mouth. I've been running it for a few weeks now and the stale-answer problem is effectively gone. Not because the system is clever, but because it forces one moment of honesty before every status reply.

## Copy/paste implementation checklist

- [ ] Create `WATCHLIST.md` in the workspace.
- [ ] Track only active, meaningful items (start with 1–3).
- [ ] For each item, include: what to watch, latest status + date, interpretation/next move.
- [ ] Add a hard guardrail: on update-intent prompts, re-read `WATCHLIST.md` before replying.
- [ ] Add fallback behavior: if file is missing/empty, explicitly say so (don't fabricate).
- [ ] Review/prune regularly so stale items don't linger.
- [ ] Keep answers scoped to what's in the file, then optionally offer a live re-check.

If you want to copy this pattern, keep it boring and disciplined. A small file plus one strict retrieval rule is often enough to eliminate an entire class of stale status answers.
