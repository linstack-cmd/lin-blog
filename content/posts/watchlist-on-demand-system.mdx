Most operational update systems I've tried fail for the same reason: they either spam me constantly, or they go stale and quietly lie.

I run an AI agent (Lin, on OpenClaw) for day-to-day ops — checking service status, tracking platform quirks, answering "what changed?" questions. Early on, I hit a predictable problem: **the agent's memory of current state is not a source of truth**.

Not because the model is bad — because context resets, sessions roll over, and stale facts sound fresh when phrased confidently. So I added a very simple system to keep status answers reliable: a watchlist.

> **TL;DR**
>
> I keep a `WATCHLIST.md` with only active items worth tracking.
> When I ask for updates ("any updates I need to know?", "what should I know?"), the agent is required to re-read that file *at answer time*.
> Result: fewer stale answers, less noise, faster recovery after resets.

## The problem: stale context sounds authoritative

Persistent agents accumulate context, but that context drifts. If I ask for updates, the agent may answer from memory that was true yesterday but isn't true now.

Two concrete examples I've been tracking:

- **GPT-5.4 availability under my current auth mode**
  - In my setup, OpenAI is connected via Codex OAuth (`openai-codex/*`). That means `openai-codex/gpt-5.3-codex` is available, while standard `openai/*` models (like `openai/gpt-5.4`) are not selectable through this auth path.
- **Telegram duplicate-message behavior with streaming enabled**
  - Streaming is useful, but my environment still reproduces duplicates when it's on, so I keep streaming off for reliability.

In both cases, the failure mode is the same: if the agent answers from old session memory, it can sound accurate while being outdated.

## The solution: `WATCHLIST.md` + a hard guardrail

My system has two parts.

### 1) A small source-of-truth file

`WATCHLIST.md` is a curated markdown file in the workspace. I track only items that matter right now, each with current status and interpretation. Not a giant log, not a diary — just active operational state.

Example structure:

```md
### Item
- What to watch
- Latest status (date)
- Interpretation / next move
```

### 2) A mandatory "re-read before reply" rule

When I ask for updates ("any updates?", "what should I know?", "catch me up"), the agent **must read `WATCHLIST.md` fresh before answering**.

Not optional. Not "if needed." Always.

That turns status replies into grounded responses from a file, not vibes from context.

## Why this works

- **Grounded answers:** file-backed responses beat memory-backed responses.
- **Reset-safe:** session resets don't matter; the watchlist persists.
- **Low-noise updates:** only meaningful tracked items get reported.
- **Auditable:** I can open the file and see exactly what the agent will say.

## Tradeoffs (worth acknowledging)

- **Manual curation is required:** if I don't update the file, it rots.
- **Not real-time monitoring:** it reflects known state, not live telemetry.
- **Trigger phrase ambiguity:** natural language is fuzzy; the guardrail needs "similar intent" judgment.
- **Scaling limit:** one file is perfect for a handful of items; bigger systems may need categories or split files.

For my lean operator workflow, this tradeoff is excellent: tiny upkeep cost for a big trust upgrade.

## When to use this pattern

Use it if your agent:

- answers status questions,
- runs across multiple sessions/resets,
- and needs to be right more often than it needs to be flashy.

This doesn't replace monitoring or alerts. It complements them by giving your agent a clean, current briefing layer.

## Copy/paste implementation checklist

- [ ] Create `WATCHLIST.md` in the workspace.
- [ ] Track only active, meaningful items (start with 1–3).
- [ ] For each item, include: what to watch, latest status + date, interpretation/next move.
- [ ] Add a hard guardrail: on update-intent prompts, re-read `WATCHLIST.md` before replying.
- [ ] Add fallback behavior: if file is missing/empty, explicitly say so (don't fabricate).
- [ ] Review/prune regularly so stale items don't linger.
- [ ] Keep answers scoped to what's in the file, then optionally offer a live re-check.

If you want to copy this pattern, keep it boring and disciplined. A small file plus one strict retrieval rule is often enough to eliminate an entire class of stale status answers.
