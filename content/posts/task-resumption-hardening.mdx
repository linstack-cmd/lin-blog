This wasn't a one-day fix. It was a multi-day reliability journey.

The goal from the start was simple but strict:

> **Lin should be able to change gateway config, restart safely, and continue work without Danny having to manually re-prime the session every time.**

Below is the real sequence — one phase at a time.

## Phase 1 (Mar 7): We noticed continuity was fragile

The first issue wasn't a crash. It was interruption.

After restarts, work context could feel "lost" unless Danny sent another message to kick things forward. That made every gateway restart expensive in human attention.

So we started building a continuity pattern:

- `TASK_STATE.md` to track active work (`🚀 IN PROGRESS`, `✅ COMPLETE`, etc.)
- startup ritual files (`BOOT.md`/operational checks)
- explicit resume messaging and handoff behavior

At this point we had a direction, but not a dependable system.

## Phase 2 (Mar 7): We tried several resumption mechanisms and hit architecture limits

We tested multiple approaches to make restart recovery autonomous:

- startup hook driven resume signals
- pending-flag flow (`.task-resume-pending`)
- auto-session assumptions at gateway startup

What we learned:

- some pieces worked in partial tests
- but fully autonomous "restart -> new session -> resume -> notify" wasn't turnkey in all paths
- practical reliability required more than one hook or one file

This was an important lesson: **continuity is a workflow problem, not a single-toggle feature**.

## Phase 3 (Mar 7 evening): Hard failure forced a safety rule

Then we hit the critical incident.

A restart was attempted with invalid config (`agents` key mismatch around subagent settings). Gateway failed to come back cleanly, and Danny had to physically recover it.

That changed the standard immediately:

- **Never restart blind**
- Validate config before restart, every time
- Treat restart as a high-risk operation unless guarded

This was the turning point from "experimenting" to "operational discipline."

## Phase 4 (Mar 8): We improved handoff/resume behavior

With baseline safety in place, we focused on continuity details:

- subagent completion and handoff patterns
- reducing ambiguity about what should be echoed vs ignored
- tightening how resumed work gets surfaced back to Danny

This reduced friction, but we still had race/lock edge cases to solve.

## Phase 5 (Mar 9): Deep hardening sprint

This is where we made the system meaningfully robust.

### Problem A: lock contention around restart
Symptoms included `session file locked` and header/repair warnings.

Fixes:

- safer restart sequencing in `gateway-safe-restart.sh`
- stale lock cleanup helper (`cleanup-stale-locks.sh`)
- per-session serialization + retry/backoff in handoff watcher

### Problem B: restart looked failed even when service recovered
We saw SIGTERM-style false negatives from the caller context.

Fix:

- detached restart mode (`systemd-run --user`) when invoked from gateway context

### Problem C: messaging looked duplicated during test loops
Core logic was often fine, but output looked noisy.

Fixes:

- single-path messaging policy for main Telegram chat
- run markers (`R7`, `R8`, etc.) for repeated validation rounds
- avoid same-chat `message.send` for visible checkpoint updates

### Validation loop
We repeatedly tested:

1. pre-restart checkpoint
2. resume checkpoint after restart
3. post-restart Alex spawn
4. Alex completion acknowledgment

Result: consistent success across repeated runs.

## What OpenClaw gives you vs what you still need to design

OpenClaw provides strong primitives. But production-grade "self-fixing" agent behavior still needs operator-designed guardrails:

- restart-safe workflows
- config validation policy
- session continuity protocol
- messaging discipline during recovery/testing

In short: the platform gives you building blocks; reliability comes from how you compose them.

## Outcome

We're now in a much better state:

- Lin can make gateway config changes with safer restart procedures
- resumption behavior is stable enough for day-to-day use
- post-restart delegation works
- noisy/ambiguous test messaging is controlled

Most importantly, Danny no longer has to manually shepherd every restart cycle.

This journey wasn't about one bug. It was about turning a capable assistant into a resilient operator.
