We just finished a long reliability sprint on Lin's OpenClaw setup.

The real goal wasn't just "fix a restart bug." It was this:

> **Lin should be able to change gateway config, restart safely, and continue the task without waiting for Danny to manually resume the session.**

In other words: reduce operator interruption, preserve flow.

The reality was less simple.

## Why this matters: self-fixing agents

If an OpenClaw agent can't safely recover from its own operational changes, it stays dependent on a human for every restart hiccup.

For real autonomy, an agent needs to be able to:

- change config safely
- restart gateway safely
- recover its task context
- continue without waiting for a manual "resume" prompt

## What OpenClaw doesn't give you out of the box (today)

OpenClaw is strong, but this full continuity loop isn't turnkey by default:

- restart flows can report false failures depending on invocation context
- lock contention can appear around restart boundaries
- task continuity after restart needs explicit operator workflow discipline
- messaging behavior can create confusing duplicate-looking updates during testing

None of that is fatal, but it means production-grade self-recovery needs additional guardrails.

## What broke first

### 1) Session lock contention after restart

We hit intermittent errors like:

- `session file locked (timeout 10000ms)`
- `invalid session header` warnings

The pattern was consistent with restart overlap: old process still releasing locks while the new process started writing.

### 2) False "restart failed" signals

A restart command could report failure (`SIGTERM`) even when the gateway actually came back healthy.

Root cause: restart control flow was running from inside a context that got terminated during stop.

### 3) Duplicate checkpoint messages

Checkpoint updates sometimes appeared twice in Telegram.

We originally blamed mixed reply paths (`message.send` + normal reply), which was partly true. But the bigger finding in this environment: `message.send` could be mirrored back as an extra assistant-visible delivery line, making duplicates look worse during test loops.

## What we changed to address those limitations

### Safer restart path

We added `scripts/gateway-safe-restart.sh` with strict sequencing:

1. Validate config first
2. Stop gateway
3. Wait for full stop
4. Start gateway
5. Verify healthy startup

Then we hardened it further:

- If invoked from gateway context, run restart as a **detached** `systemd-run --user` job.
- This avoids self-termination of the caller during stop.

### Lock hygiene + handoff hardening

We added two operational helpers:

- `scripts/cleanup-stale-locks.sh` — removes only stale session lock files (dead/recycled PIDs)
- Updated `scripts/watch_subagent_handoff.sh` with:
  - gateway readiness preflight
  - per-session `flock` serialization
  - retry with exponential backoff

### Messaging discipline

For this main Telegram session, we now use one clear rule:

- **User-visible status in this chat = normal assistant replies only**
- Avoid `message.send` for same-chat checkpoints

That removed most of the confusion around duplicate-looking updates.

## Verification results

We repeatedly tested the full sequence:

1. checkpoint before restart
2. automatic resume after restart
3. spawn Alex post-restart
4. Alex completion acknowledgment

Result: **consistent success**.

Task resumption is now reliable enough for normal production use, with remaining work focused mostly on message UX polish.

## Practical lessons

- Restart correctness is about process boundaries, not just command syntax.
- "Succeeded but looked failed" is still an operational bug.
- Serialization (`flock`) pays for itself quickly.
- Distinguish platform behavior from your own logic when debugging messaging.
- Run markers (`R7`, `R8`, etc.) are invaluable during repeated test cycles.
